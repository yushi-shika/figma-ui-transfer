# 解説: FigmaのデザインからアプリのUIを自動生成する実験 — 何をどうやったのか

このドキュメントは、エンジニアではない方向けに「今回の実験で何をやったのか」を説明するものです。専門用語はできるだけ噛み砕いて書いています。

## 一言でいうと

**「デザインツール(Figma)で描かれたアプリの見た目を、AIに読み取らせて、3種類の技術で実際に動くアプリ部品に自動変換させた。そして、どの技術が一番効率よく・正確に変換できるかを比べた」** という実験です。

さらにおまけとして、**逆方向(書いたプログラム → Figmaのデザインデータ)への変換**もできるかを試しました。

## 登場するツール(それぞれ一言で)

| ツール | 何をするもの? |
|---|---|
| **Figma** | デザイナーがアプリやWebの見た目を描くための、業界標準のデザインツール |
| **Claude Code** | 会話で指示すると、プログラムを書いたり作業したりしてくれるAIアシスタント |
| **MCP(Model Context Protocol)** | AIが外部のツール(今回はFigma)と直接やり取りするための「共通コンセント」のような仕組み。今回はこれでAIがFigmaの中身を直接読めるようにした |
| **Flutter / React / React Native** | アプリの画面を作るための代表的な技術3種。会社やプロジェクトによって採用している技術が違うため、3つとも試した |
| **Storybook / Widgetbook** | 作ったボタンや入力欄などの「部品」を、カタログのように一覧表示して確認できるツール |
| **GitHub** | プログラムの保管・共有場所 |
| **Vercel** | 作ったものをURL一つで誰でも見られるように公開できるサービス |

## 元にしたデザイン

Figmaの公開デザイン集(Community)にある「Shopping App UI Kit」という、靴のショッピングアプリのデザイン(全10画面)を使いました。

- 元デザイン: https://www.figma.com/community/file/1005735282905530510
- 複製して実際に読み取ったファイル: https://www.figma.com/design/cobQGyPNOOx3MyrDicL75d/Shopping-App-UI-Kit-1.0--Community-

※Figmaの公開デザインは、一度自分のアカウントに複製しないとAIから読み取れない仕様のため、複製してから使っています。

## 実際の流れ(7ステップ)

### ステップ1: AIとFigmaをつなぎ、アカウントを確認する

まず、Claude Code(AI)がFigmaに接続できているかを確認しました。接続確認と同時にアカウントの契約状態も分かるのですが、ここで重要な制約が判明しました。

> **使っていたFigmaアカウントは無料相当のプラン(View権限)だったため、AIからFigmaのデータを読み取れるのは「月に6回まで」だった。**

この制約が、以降の進め方すべてに影響しています。

### ステップ2: デザイン全体の「目次」を1回で取得する

貴重な6回の枠を無駄にしないため、まず1回の読み取りで「このデザインファイルには何の画面が何枚あるか」という目次情報だけを取得しました(サインアップ画面、ログイン画面、商品一覧、購入画面…全10画面あることが判明)。

### ステップ3: 2画面だけを選んで、詳細データを取得する(ここで計3回消費)

10画面すべてを読むと枠が足りないので、部品が豊富な**「サインアップ画面」と「商品詳細画面」の2つだけ**を選び、色・サイズ・配置・文字などの詳細データを取得しました。

そして取得したデータから、多くのアプリで使い回せる代表的な部品を3つ抜き出しました:

- **ボタン**(「Create Account」「Buy for $338」など)
- **入力欄**(名前・メール・パスワードの記入欄)
- **商品カード**(靴の写真+商品名のセット)

**ポイント**: ここで取得したデータはすべて手元にファイルとして保存し、**以降の作業ではFigmaに二度とアクセスしない**方針にしました。月6回制限の中で実験を成立させるための工夫です(結果的に使ったのは6回中4回)。

### ステップ4: 3つの技術に共通の「変換ルールブック」を作る

3つの技術で公平に比較するため、「この色はこの用途に使う」「角の丸みは6ピクセル」など、デザインから読み取った数値を1冊のルールブック(文書)にまとめました。3チームに同じ設計図を渡すイメージです。

### ステップ5: AIを3人に分身させて、同時に3つの技術で実装する

Claude Codeの機能で、**独立した3つのAI作業者を同時に走らせました**。それぞれがFlutter版・React版・React Native版を担当し、お互いの作業は見えない状態(カンニング禁止)で、同じルールブックとデザインデータだけを頼りに部品を実装しました。

結果、3つとも自動チェック(文法検査・ビルド・テスト)にすべて合格。かかった体感時間は React Native(約30〜40分)< React(約60〜75分)< Flutter(最長・一度中断あり)でした。

### ステップ6: 部品を組み合わせて「本物の画面」を1枚作る

部品単体のカタログだけでは分かりにくい、という指摘を受けて、ステップ3で保存済みのデータを再利用し(Figmaへの追加アクセスなし)、**「サインアップ画面」を3つの技術すべてで1枚の完成画面として組み立てました**。背景写真・ロゴ・入力欄3つ・ボタン・注意書きまで、元のFigmaデザインとほぼ同じ見た目を再現できています。

### ステップ7: 逆方向(プログラム → Figma)も試す

最後に、React版で作った画面を**逆にFigmaへ取り込む**機能を試しました。ブラウザに表示した画面をそのままFigmaのデザインデータに変換する仕組みで、こちらは回数制限の対象外だったため無料プランでも実行できました。

- 取り込み結果: https://www.figma.com/design/zd6oOjMxLqgzE1Y667sPeB?node-id=1-2

見た目はきれいに再現されましたが、「これはボタンである」「これは入力欄である」といった**意味の情報は失われ、ただの図形として取り込まれる**ことも分かりました(想定通りの結果です)。

## わかったこと

1. **AIはFigmaのデザインを読み取って、3種類の技術どれでも、ほぼ忠実にアプリ部品へ変換できる。** 人間が書き直す必要はほとんどなかった。
2. **技術による差は「正確さ」より「手間」に出る。** React Nativeが最も速く、Flutterはセットアップの手順が多く時間がかかった。
3. **プログラム→Figmaの逆変換も「見た目の再現」なら可能。** ただしデザイン部品としての意味情報は失われる。
4. **Figmaの契約プランが実務上のボトルネックになる。** 無料相当プランでは月6回しか読み取れないため、本格運用にはプランのアップグレード(有料のFull/Devシート)が必要。

## 今回できなかったこと(プラン制約のため)

- AIがFigma上に直接デザイン部品を「新規作成」する機能(有料のFullシートが必要)
- プログラムとFigmaのデザイン部品を正式に紐付ける「Code Connect」機能(法人向け上位プランが必要)

これらは、プランをアップグレードすれば追加検証できる状態です。
